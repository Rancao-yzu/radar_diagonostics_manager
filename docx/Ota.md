# OTA 升级  
- 提供 Hex 文件  
- OTA 升级方案基于 PFBoot 实现客户 App 的升级  
- 芯片上电后依次执行两级 Boot，每一级 Boot 完成自身职责后跳转到下一级，最终进入客户应用 CUAP  
- OTA 升级流程中，上位机发送 FORCEJUMP 安全帧，BM 识别后将停留在 PFBoot 不再跳转至 APP，等待接收 UDS 刷写命令  

---

## X   软重启
#### X1：扩展会话切换
- 请求 [X1] 02 10 03
- 响应 50 03 ...
  
02 = 数据长度 2 字节(有效数据长度)

#### X2：ECU复位
- 请求 [X2] 02 11 01
- 响应 51 01
  
ECU 复位中 100-200ms

> X过程为模拟对雷达进行软重启，该过程为可选，根据实际情况是否需要。如不需要，烧录前应该对雷达进行硬重启（重新上电）。


## A 发送安全帧（触发 PFBoot OTA 模式）  
- 上位机以 50 ms 周期发送专用 CAN-FD 扩展帧 ID `0x190C8532`，共发送 100 次。  
- 数据负载中包含 `FORCEJUMP` 字符串与 `0xA5B6C7D8`，用于通知 BM 停留在 PFBoot。  

```example
Tx/Rx,ID,DLC,MSG,STATUS
tx,190C8532,16,02 10 60 46 4F 52 43 45 4A 55 4D 50 A5 B6 C7 D8,OK
```

## D 安全访问（Security Access）  
- 安全等级 `0x61`，通过 DLL 完成种子请求与密钥计算。  
- 实际传输使用 CAN‑FD 多帧，种子与密钥长度可能较大，以下为真实报文示例。

**1. 请求种子**  
`02 27 61`

```example
8.6996, tx, 74C, 8, 02 27 61 00 00 00 00 00, OK
```

**2. 接收种子（多帧响应）**  
- 首帧 `10 FC 67 61 00 19 01 F4 ...` 表示种子数据总长度 252 字节，首帧携带 `67 61` 正响应及种子前部。  
- 上位机发送流控帧 `30 08 00`，ECU 随后以连续帧 `21, 22, 23, 24` 发送剩余种子数据。

```example
rx, 7CC, 64, 10 FC 67 61 00 19 01 F4 00 00 ... , OK  // 首帧
rx, 7CC, 64, 21 00 00 00 00 ... , OK                // 连续帧 1
rx, 7CC, 64, 22 00 00 00 00 ... , OK
rx, 7CC, 64, 23 00 00 00 00 ... , OK
rx, 7CC, 8,  24 00 00 00 00 00 00 00, OK            // 最后一帧
```

**3. 发送密钥**  
- 上位机根据 DLL 计算密钥（此处示例密钥全为 `0x11`），组装 `27 62` + 密钥数据。  
- 首帧 `17 D2 27 62 11 11 ...`，数据总长度 2002 字节，随后通过流控与连续帧完成全部发送。

```example
tx, 74C, 64, 17 D2 27 62 11 11 ... , OK  // 首帧，密钥数据开始
tx, 74C, 64, 21 11 11 11 11 ... , OK       // 连续帧 1
... (连续帧 22 ~ 2F，每帧承载密钥数据)
tx, 74C, 64, 2F 11 11 ... 00 00 00, OK   // 最后一帧
```

**4. 安全访问确认**  
- ECU 回复 `67 62`，表示密钥验证通过。

```example
8.7282, rx, 7CC, 8, 02 67 62 00 00 00 00 00, OK
```

## E 擦除目标区域（APP）  
- 请求 `31 01 FF 00` + 4 字节起始地址 + 4 字节长度  
  （地址和长度由 Hex 文件决定，本例中地址 `0x00 09 00 00`，长度 `0x00 00 20 00`）  
- 擦除期间 ECU 返回 `7F 31 78` 表示请求已接收，正在处理，需持续等待。  
- 最终正响应 `71 01 FF 00 10 00 00` 表示擦除完成。

```example
tx, 74C, 16, 00 0C 31 01 FF 00 00 09 00 00 00 20 00 00 00 00, OK   // 发送擦除请求
rx, 7CC, 8,  03 7F 31 78 00 00 00 00,                      OK   // 延迟响应 1
rx, 7CC, 8,  03 7F 31 78 00 00 00 00,                      OK   // 延迟响应 2
rx, 7CC, 8,  05 71 01 FF 00 10 00 00,                      OK   // 擦除完成
```

---

## F 请求下载  
- 请求 `34 00 44` + 4 字节起始地址 + 4 字节长度  
  （地址和长度由 Hex 文件决定，本例中地址 `0x00 09 00 00`，长度 `0x00 00 20 00`）  
- 正响应 `74 20` + MaxBlockSize，表示下载请求接受，最大数据块为 4095 字节（`0x0F FF`）。

```example
tx, 74C, 16, 00 0B 34 00 44 00 09 00 00 00 20 00 00 00 00 00, OK   // 请求下载
rx, 7CC, 8,  04 74 20 0F FF 00 00 00,                      OK   // 正响应
```

## G 传输数据  
- 使用 TransferData 服务（`36`），每个数据块包含一个序列号（SN），从 `01` 开始递增。  
- 首帧格式为 `1F FF 36 SN [data...]`，`1F FF` 为 CAN‑FD 首帧标志，表示后续数据长度较大。  
- ECU 回复流控帧 `30 00` 后，上位机以连续帧 `21`、`22`... 依次发送剩余数据，直至最后帧。  
- ECU 可能先返回 `7F 36 78` 延迟响应，然后返回 `76 SN` 确认该块接收成功。

示例（SN=0x01）：
```example
tx, 74C, 64, 1F FF 36 01 [data...], OK           // 首帧，携带数据块 01 开头
tx, 74C, 64, 21 [data...], OK                     // 连续帧 1
... (连续帧 22 ~ 2F，随后循环 20、21... 直至数据发完)
tx, 74C, 8,  21 84 00 00 00 00 00 00, OK         // 最后一帧
rx, 7CC, 8,  03 7F 36 78 00 00 00 00, OK         // 延迟响应（ECU 正处理）
rx, 7CC, 8,  02 76 01 00 00 00 00 00, OK         // 块 01 接收成功
```

> **说明**：实际传输中多个数据块依次发送（SN=01, 02, ...），最后一包数据完成后进入传输退出流程。

## H 传输退出  
- 所有数据块发送完毕后，上位机发送 `01 37` 请求退出传输。  
- ECU 回复 `77` 表示传输退出成功。

```example
tx, 74C, 8, 01 37 00 00 00 00 00 00, OK
rx, 7CC, 8, 01 77 00 00 00 00 00 00, OK
```

## I CRC 校验  
- 请求 `31 01 02 12` + 4 字节 CRC32（由 Hex 文件提供）  
- 正响应 `71 01 02 12 10 00`  

```example
tx, 74C, 12, 00 08 31 01 02 12 23 E1 6E BA 00 00, OK   // 请求 CRC 校验
rx, 7CC, 8,  03 7F 31 78 00 00 00 00,        OK   // 延迟响应 1
rx, 7CC, 8,  06 71 01 02 12 10 00 00,        OK   // 校验通过
```

## J 写客户标志（CUAP）  
- 请求 `2E FC 01 43 55 41 50`（"CUAP" 的 ASCII）  
- 响应 `6E FC 01`  

```example
tx, 74C, 8, 07 2E FC 01 43 55 41 50 00, OK
rx, 7CC, 8, 03 6E FC 01 00 00 00 00,    OK
```

## K 依赖检查（Dependency Check）  
- 请求 `31 01 02 05`  
- 响应 `71 01 02 05 00`  

```example
tx,74C,8,04 31 01 02 05 00 00 00,OK
rx,7CC,12,00 09 71 01 02 05 10 00 00 00 00 00,OK
```

## L ECU 复位  
- 请求 02 10 01 使 ECU 复位，进入新刷写的应用程序。

- ECU 可能回复 7F 10 78（延迟响应），表示复位正在执行中，随后完成复位（可能不显示）

```example
63.0002, tx, 74C, 8, 02 10 01 00 00 00 00 00, OK   // 请求复位
63.0025, rx, 7CC, 8, 03 7F 10 78 00 00 00 00, OK   // 复位进行中
```
